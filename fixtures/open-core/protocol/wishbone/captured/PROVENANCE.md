# Captured fixtures — Wishbone

This directory holds traces acquired or produced from public open-source
projects. They complement the deterministic synthetic fixtures next door in
`../generated/` by exercising the decoder against real-world bus traffic.

## License allow-list

Captured fixtures must come from projects under one of these licenses:

- MIT
- BSD-2-Clause / BSD-3-Clause
- Apache-2.0
- ISC
- CC0-1.0
- public domain

GPL/AGPL/proprietary captures are excluded by policy. The static test
`test/static/captured_fixture_licenses_test.dart` walks every
`PROVENANCE.md` in the corpus and fails the build if any entry uses a
license outside this list, or if any fixture file in this directory has no
matching entry.

## Per-fixture entries

Each fixture committed below carries a block in this file with: source
project, source commit SHA, license + SPDX, author/maintainer, original
location of the testbench inside the source repo, capture method (direct
file from repo, or the exact `iverilog` / `verilator` / `cocotb`
invocation used locally), acquisition date, file size, scenario summary,
and 2–3 hand-verified anchor transactions confirmed by eye before the
snapshot was committed.

### Block template

```markdown
## `<fixture_name>.fst` — <one-line scenario summary>

- **Source project:** <project name> — <https://github.com/...>
- **Source commit:** `<sha>` (acquired YYYY-MM-DD)
- **License:** <name> (SPDX: `<identifier>`)
- **Author / maintainer:** <name>
- **Original location:** `<path/to/testbench>`
- **Capture method:** <"direct .fst from repo" | "built locally with `iverilog …`" | "built locally with `verilator --trace-fst …`" | "cocotb run: `pytest tb/… `">
- **Acquired:** YYYY-MM-DD
- **Size:** <bytes raw / bytes committed>
- **What it exercises:** <2–3 sentence scenario description>
- **Hand-verified anchors:**
  - <e.g. "write to 0x40, data 0xdeadbeef, at t≈230 ns">
  - <e.g. "burst of 4 reads starting at 0x1000">
- **Snapshot regenerated by:** `dart run tool/regenerate_captured_snapshots.dart --decoder wishbone --fixture <fixture_name>`
```

## Current captures

## `wishbone_picorv32_wb_ez.fst` — RISC-V ifetch + load/store loop on picorv32_wb

- **Source project:** PicoRV32 — <https://github.com/YosysHQ/picorv32>
- **Source commit:** `87c89acc18994c8cf9a2311e871818e87d304568` (acquired 2026-05-31)
- **License:** ISC (SPDX: `ISC`)
- **Author / maintainer:** Claire Xenia Wolf
  (<claire@yosyshq.com>) — YosysHQ
- **Original location:** `picorv32.v` (module `picorv32_wb`).
  The hand-encoded RV32I loop body is lifted verbatim from
  upstream `testbench_ez.v` (public-domain section of the same
  repo).
- **Capture method:** `cd helpers/picorv32 && make install`
  (vendored `picorv32.v` + our `tb_picorv32_wb_ez.v` testbench,
  built with `iverilog -g2012` and dumped via
  `vvp tb_picorv32_wb_ez.vvp -fst`). No RISC-V toolchain — the
  program is six hand-encoded RV32I instructions sitting in the
  testbench's WB-slave memory.
- **Acquired:** 2026-05-31
- **Size:** ~5.7 kB committed
- **What it exercises:** Ten Wishbone B3 Classic transactions
  produced by letting `picorv32_wb` fetch and execute a tight
  counter loop (`li x1, 1020` / `sw x0, 0(x1)` / `lw x2, 0(x1)` /
  `addi x2, x2, 1` / `sw x2, 0(x1)` / `j loop`). The trace mixes
  instruction-fetch reads (`sel=0x0`), data-load reads, and
  full-word data stores (`sel=0xF`) — exactly the kind of
  composite RISC-V-on-WB traffic that the synthetic
  generated/ fixtures don't cover. Single-cycle ack on every
  transfer; no errors or retries.
- **Hand-verified anchors:**
  - `R 0x00000000 → 0x3FC00093` at t=80 ns — first ifetch is the
    `li x1, 1020` instruction word (0x3FC00093).
  - `W 0x000003FC = 0x00000000` at t=250 ns — counter init store
    (the `sw x0, 0(x1)` at the head of the loop).
  - `W 0x000003FC = 0x00000001` at t=530 ns — second counter
    store, proving the increment (`addi x2, x2, 1`) executed
    and the loop branched back.
- **Snapshot regenerated by:** `REGENERATE=1 flutter test test/services/decoders/wishbone_captured_fixtures_test.dart`
