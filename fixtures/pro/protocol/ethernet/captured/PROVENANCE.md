# Captured fixtures — Ethernet (MII / RMII / GMII / RGMII / AXI-Stream)

This directory holds traces acquired or produced from public open-source
projects. They complement the deterministic synthetic fixtures next door in
`../generated/` by exercising the decoder against real-world bus traffic.

## License allow-list

Captured fixtures must come from projects under one of these licenses:

- MIT
- BSD-2-Clause / BSD-3-Clause
- Apache-2.0
- ISC
- CC0-1.0
- public domain

GPL/AGPL/proprietary captures are excluded by policy. The static test
`test/static/captured_fixture_licenses_test.dart` walks every
`PROVENANCE.md` in the corpus and fails the build if any entry uses a
license outside this list, or if any fixture file in this directory has no
matching entry.

## Per-fixture entries

Each fixture committed below carries a block in this file with: source
project, source commit SHA, license + SPDX, author/maintainer, original
location of the testbench inside the source repo, capture method (direct
file from repo, or the exact `iverilog` / `verilator` / `cocotb`
invocation used locally), acquisition date, file size, scenario summary,
and 2–3 hand-verified anchor transactions confirmed by eye before the
snapshot was committed.

- **Source project:** alexforencich/verilog-ethernet — <https://github.com/alexforencich/verilog-ethernet>
- **Source commit:** `77320a9471d19c7dd383914bc049e02d9f4f1ffb` (acquired 2026-05-30)
- **License:** MIT (SPDX: `MIT`) — per-file header in `tb/eth_mac_1g_rgmii/Makefile` and `tb/eth_mac_1g_rgmii/test_eth_mac_1g_rgmii.py`
- **Author / maintainer:** Alex Forencich
- **Original location:** `tb/eth_mac_1g_rgmii/test_eth_mac_1g_rgmii.py` + `rtl/eth_mac_1g_rgmii.v`
- **Capture method:** cocotb run from a clean clone of upstream — `cd tb/eth_mac_1g_rgmii && patch -p1 < helpers/forencich_eth_mac_1g_rgmii_cocotb2_1gbe.patch && PATH=~/.venvs/wavecrux-cocotb/bin:$PATH make WAVES=1 SIM=icarus` (see [`../../axi4_full/captured/helpers/cocotb_setup.md`](../../axi4_full/captured/helpers/cocotb_setup.md) for the venv + simulator setup, and [`helpers/README.md`](helpers/README.md) for the Ethernet-specific deltas)
- **Acquired:** 2026-05-30
- **Size:** 52978 bytes raw FST (well under 5 MB target; snapshot JSON is ~30 KB)
- **What it exercises:** 4 cocotb test scenarios concatenated into one FST — `run_test_rx_001` (80 incrementing-payload frames sized 60..69 B + 512 B + 1514 B fed in via cocotbext-eth's `RgmiiPhy.rx.send`), `run_test_tx_001` (same frame set looped back through the MAC TX path), `run_test_tx_underrun_001` (3 frames, deliberate AXIS stall to provoke the MAC's underrun-error injection), `run_test_tx_error_001` (3 frames with `tuser=1` on the middle frame to inject a TX error). Bound to the **RX** direction here, so only the `run_test_rx_001` window (0..24468 ns) produces frame transactions — the TX windows are silent on `rgmii_rx_ctl`. A future `_tx` companion fixture (same FST, separate `.fixture.json`) would exercise the TX direction.
- **Hand-verified anchors:**
  - First RX frame at t=852000 ps decodes as `dst_mac=00:01:02:03:04:05`, `src_mac=06:07:08:09:0A:0B`, `ethertype=0x0C0D`, `frame_size=64` (60 B payload + 4 B FCS) — matches the testbench's `bytes(x for x in range(60))` incrementing payload generator and `GmiiFrame.from_payload(test_data)` framing.
  - Frame 11 in the snapshot is the 512-byte mid-sweep frame; frame 12 is the 1514-byte jumbo-boundary frame. Both decode with the same incrementing-payload pattern at the expected byte counts.
  - No frames decoded after t≈24468 ns: the TX windows (`run_test_tx_001`, `run_test_tx_underrun_001`, `run_test_tx_error_001`) drive `rgmii_txd`/`rgmii_tx_ctl`, not the RX side this fixture is bound to.
- **Snapshot regenerated by:** `REGENERATE=1 flutter test test/services/decoders/ethernet_rgmii_captured_fixtures_test.dart`

### Finding — resolved in the companion commit

When the FST first landed, the decoder mis-flagged every frame as
`rgmii_rx_er asserted mid-frame` because the RGMII RX_ER detection in
`lib/services/decoders/ethernet/ddr_stream_walker.dart` had its
negedge-`ctl` polarity inverted relative to the RGMII v2.0 spec:

- Per spec: at negedge clock, `rgmii_rx_ctl = RX_DV XOR RX_ER`. During
  a valid frame (`RX_DV=1`), `RX_ER=0` ⇒ negedge `ctl=1` (no error);
  `RX_ER=1` ⇒ negedge `ctl=0` (error).
- Decoder pre-fix: latched `errorAsserted=true` when negedge `ctl=1`,
  which is the **no-error** wire condition. The synthetic generator at
  `tool/ethernet_rgmii_sim.dart` was authored to the same wrong
  convention, so the `generated/ethernet_rgmii_basic`, `_errors`, and
  `_jumbo` snapshots agreed with the buggy decoder and stayed green —
  the inversion was completely invisible inside the synthetic corpus.

cocotbext-eth's `RgmiiPhy` and Forencich's `rgmii_phy_if.v` both follow
the actual spec, which is why this captured fixture surfaced the
inversion that the synthetic corpus had been masking. The fix (one
boolean inversion in the walker + a matching encoder fix in the
synthetic generator + regeneration of all four RGMII snapshots) lands
in the immediately-following commit. The snapshot in this directory
reflects the **corrected** decode: 12 clean transactions, no spurious
errors.
